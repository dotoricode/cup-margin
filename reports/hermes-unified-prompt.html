5.작업 원칙

  1. 백로그 ID = PR 단위. P0는 각각 별도 PR. P1·P2는 코드 영역이 같으면 묶기 가능.
  2. 카피는 model/copy.ts에 상수로. 컴포넌트 인라인 하드코딩 금지.
  3. 색상은 디자인 토큰으로. text-[#64748d] 같은 Tailwind 임의값 금지. CSS 변수 또는 tailwind config에 text-muted, bg-soft 등 의미 토큰으로 추출.
  4. TDD. 도메인 로직(마진 구간별 등급, 입력 신뢰도 계산, 의사결정 카피 매핑)은 Vitest로 RED→GREEN→REFACTOR.
  5. 불변 패턴. 기존 객체 변형 금지, 새 객체 반환.
  6. 사장님 언어로만 카피 작성. 토스의 "이해하기 쉬운 용어" 원칙. 영어 SaaS 직역 금지.
  7. 의미 시맨틱. div 더미 대신 main/section/aside/nav. axe-core landmark-complementary-is-top-level 위반은 이 원칙에서 발생.
  8. 성능 우선 로딩. 차트·CSV·공유 모달은 next/dynamic. 폰트는 next/font로 자체 호스팅, 핵심 1종만 preload.
  9. R1·R4 강점은 절대 건드리지 말 것. 두 검수가 합의한 "유지" 항목.
  10. 데이터 처리 약속 보존. "계산 결과는 이 기기에만 저장됩니다"는 P1 신뢰의 핵심. 백엔드 API 도입 시도 동의 없이 매장 데이터 서버 전송 금지.

# 가설 검증: 페이지 RSC인지, 어디서 클라이언트 트리가 시작되는지

2단계 가설

  • 차트·CSV·공유·모달은 사용자 행동 후 로드되어도 됨 → next/dynamic({ ssr: false }).
  • 폰트가 외부 CDN이면 → next/font 자체 호스팅, 핵심 1종만 preload.

9.카피 변경 규칙

  • 두 검수의 카피 개선안 중 겹치는 것은 그대로 채택. Audit B 단계 7의 4개 화면 카피는 카피 라이브러리의 기본값.
  • 베타 사용자 수·평균 이익률은 실 데이터로 치환. 가짜 숫자 절대 금지. 실 데이터가 작으면 정직하게 "베타 사용자 N명".
  • "카페 평균 55%대"는 P2-12 페이지 완성 전까지 각주 보류. P1-03 PR에서는 상수만 박고 "출처 페이지 준비 중" 처리.
  • 이모지(💚, 📍)는 디자인 리뷰 후 결정. 첫 PR은 텍스트만으로 동일 의미 전달 가능한지 확인.
  • 결과 보조 라벨 색은 P0-13에서 새로 만든 text-muted 토큰을 사용. P1-07 PR에서는 신규 텍스트만 추가, 색 토큰은 토큰 시스템에서 가져옴.

12.절대 하지 말 것

  • 두 검수 백로그에 없는 기능 추가 (회원가입, 결제, AI 추천). 베타 단계 R2 강점 보존.
  • 영문 SaaS 카피 직역 ("Get started", "Powerful insights" 등).
  • "양호" 한 단어로 끝나는 결과 카피 부활. P1-03이 정확히 그걸 고치는 작업.
  • 검수 결과를 임의 재해석해 백로그 스킵. 스킵하려면 사용자에게 근거와 함께 묻기.
  • P0-14를 P0-04 없이 시작. 측정 가드레일 없이 LCP 회복 작업은 회귀 추적 불가.
  • P0-14 phase 사이에 LHCI 측정 건너뛰기. 각 phase 후 mobile-calc Perf·LCP를 PR에 표로 기록.
  • 색대비 위반을 "회색 한 톤만 어둡게" 식으로 즉흥 픽스. 토큰 1개로 통일하지 않으면 다음 PR에서 회귀.
  • P1-11에서 카피와 코드 동작이 다른데 카피만 바꾸기. 데이터 전송 약속과 실제 구현이 일치해야 함 — 거짓 약속 절대 금지.
  • 샘플 9개 메뉴를 완전 제거. P0-01은 "기본은 1개, 9개는 보조 버튼 뒤로"이지 삭제가 아님.
  • --no-verify로 git hook 건너뛰기. 빌드/lint/test 실패는 다음 항목으로 넘어가기 전에 해결.
